# Agent Game Team Prototype Spec

この文書は、`Agent-Lab/Agent-team/tools/agent-game-team` を、`game-jinja` のゲーム制作プロトタイプとして取り込むための仕様書です。

## 目的

ローカル LLM のエージェントチームが、仕様、実装、テスト判定を分担しながら、ブラウザで遊べる 1 ファイル HTML ゲームを反復制作する。

この repo では、その仕組みを発展させて「ゲーム神社」のプロトタイプ制作ラインにする。

## 背景

`agent-game-team` は 2026-06-13 の DGX Spark ハッカソン向けに作られた、テスト駆動のゲーム制作装置である。

主観的な critic は置かず、`game-spec.md` の `## tests` に書かれた客観的な合格条件を完成ゲートにする。2026-06-12 の試走で、critic が実装済み機能を「無い」と判定し続ける問題があったため、完成判定はテスト条件に寄せる。

## チーム構成

| 役割 | 想定モデル | 責務 |
| --- | --- | --- |
| ディレクター | 人間 | 意図、優先順位、裁定を入力する |
| オーケストレーター | `gpt-oss:120b` | テスト結果を見て、実装修正か仕様修正かを切り分ける |
| デザイナー | `gpt-oss:20b` | `game-spec.md` に仕様と合格テストを書く |
| ビルダー | `gpt-oss:20b` | 仕様を満たす `index.html` を実装する |
| ランナー | `gpt-oss:20b` | 各テストをコード読解で pass/fail 判定する |

## 生成物

ゲームごとに 1 ディレクトリを生成する。

```text
games/<YYYYMMDD-HHMMSS>-<title-slug>/
├── index.html      # ゲーム本体。単一 HTML ファイル
├── game-spec.md    # 仕様 + ## tests。SSOT
└── bus.jsonl       # 制作会議の全ログ
```

`state/current.txt` は現在制作中のゲームディレクトリを指す。

## 制作ループ

1. ディレクターが作りたいゲームの brief を入力する。
2. デザイナーが `game-spec.md` を作る。
3. デザイナーは `## tests` に 4〜6 個の客観的な合格条件を書く。
4. ビルダーが `index.html` を単一ファイルで作る。
5. ランナーが `## tests` を 1 件ずつ pass/fail 判定する。
6. オーケストレーターが失敗原因を分類する。
7. 実装バグならビルダーに戻す。
8. 仕様やテスト条件の問題ならデザイナーに戻す。
9. 全テスト pass で完成宣言する。

## Cockpit

`cockpit/server.py` は制作状況を見るためのローカル司令室である。

表示領域:

| 領域 | 内容 |
| --- | --- |
| 司令塔 | ディレクターとオーケストレーターの発言 |
| 議論 | デザイナー、ビルダー、ランナーの発言 |
| 仕様書 + テスト | 現在の `game-spec.md` |
| ゲーム本体 | 別タブで `index.html` を開く |

ディレクター入力は `POST /api/director` 経由で `bus.jsonl` に追記される。

## 実行イメージ

```bash
# 司令室
python3 cockpit/server.py

# 新規ゲーム制作
python3 game_team.py "祈りの言葉と選んだ遊びの気配から、その場で遊べる短いブラウザゲームを授ける" --max-turns 5

# 追加指示を現在のゲームへ反映
python3 game_team.py --continue --max-turns 3

# current ポインタをクリア
python3 game_team.py --reset
```

## モデル設定

デフォルト:

```text
OLLAMA_URL=http://localhost:11434
ORCH_MODEL=gpt-oss:120b
WORK_MODEL=gpt-oss:20b
```

DGX Spark 119GB を想定し、120b と 20b を同時利用する構成。

## ゲーム神社への発展方針

この repo で作るゲームは、`agent-game-team` の汎用ゲーム生成ラインを、祈りと遊びの気配からミニゲームを生成する体験に寄せていく。

初期プロトタイプの対象:

| 項目 | 方針 |
| --- | --- |
| 入力 | 最大 7 つの遊びの気配 + 短い祈りの言葉 |
| 神託生成 | 入力をもとにゲームコンセプト、ルール、操作、スコア、ペナルティを決める |
| ゲーム生成 | その場で遊べる短いミニゲームを単一 HTML として生成する |
| 画面 | 祈りの入力、神託表示、ゲーム本体を 1 つの体験として扱う |
| 保存 | 最初は生成ログのみ。必要になったら localStorage や JSON export/import |
| テスト | 仕様書の `## tests` に客観判定できる条件として明記 |

## 最初の game-spec.md に入れるべきテスト例

```text
T1: 遊びの気配を最大 7 つまで選択でき、8 つ目は選択されない。
T2: 祈りの言葉を入力して「祈る」を押すと、神託としてゲームタイトル、ルール、操作説明が表示される。
T3: 神託後に、そのまま遊べるゲーム画面が表示される。
T4: ゲームにはスコア、失敗条件、リスタート操作が存在する。
T5: リスタート操作でスコア、状態、勝敗表示が初期状態に戻る。
```

## 現時点の制約

- 生成ゲームは外部ライブラリなしの単一 `index.html` とする。
- ランナーの判定は現時点ではコード読解ベースであり、実ブラウザ操作ではない。
- 生成 AI の本番接続は初期版では必須にしない。まずは固定ロジックやローカル LLM で神託体験を固定する。
- 画像、音、サーバー保存は初期プロトタイプの完成条件に含めない。

## 次の改善候補

- Playwright で実ブラウザ E2E テストを行う。
- 遊びの気配の選択 UI を増やし、生成されるゲームジャンルの幅を広げる。
- 神託文とゲームルールの一貫性をテスト条件に含める。
- 司令室からの追加指示を会議ループ中にリアルタイム反映する。
- 完成したゲームを `examples/` に固定保存し、失敗時の比較対象にする。
