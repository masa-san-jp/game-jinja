## 0. ハード制約（マサさん指定・絶対）

| # | 制約 | 根拠/影響 |
|---|------|----------|
| C1 | **入力はゲームパッド的操作のみ**：カーソルキー（↑↓←→）＋ `Z` ＋ `C`。テキスト入力は一切なし | 神社の選択・祈りも全部キー操作で完結させる |
| C2 | **ゲームが出るまで 1 分以内** | 生成戦略の中核制約（§6）。LLM 多ターン生成は間に合わないため事前生成プールを基本とする |
| C3 | **1 プレイ 3 分以内に必ず終わる** | 全ゲームに制限時間 or 明確な終了条件＋結果画面を必須化 |
| C4 | ゲームはシンプル（1 画面・短時間で理解できる） | 来場者が説明なしで遊べる |

## 1. 目的とゴール

- **デモのゴール**：来場者がタイトルから始め、2 択の質問に答え、「祈る」と、AI（DGX Spark のローカル LLM）が作ったゲームを授かり、1 分以内にその場で遊べる。
- **見せ場**：2 択＝「神様への願い」、生成AI＝「神託（授かりもの）」という演出。
- **非ゴール**：認証・課金・クラウド永続保存・3D・同時対戦・テキスト入力。

## 2. 体験フロー（マサさん指定）

1. **タイトル画面**：「ゲーム神社」。`ゲームを始める`（決定 = `Z`）。
2. **2 択の質問**：いくつかの質問が順に出る。各質問は 2 択（←→ で選び `Z` で決定）。答えが「欲しいゲームの気配」になる。
3. **祈り**：`あなたが欲しいゲームを授けます。祈ってください。`＋神社モチーフ（鳥居・賽銭・しめ縄）。`Z` 長押し or 連打で「祈る」演出。
4. **授与（生成）**：1 分以内にゲームが表示される（§6 の戦略で担保）。待ち演出は最大でも数十秒。
5. **プレイ**：カーソル＋`Z`＋`C` で遊ぶ。3 分以内に終了し結果画面（スコア/クリア/タイムアップ）。
6. **戻る**：タイトルへ（`C` で戻る等）。遊んだゲームはセンター一覧に蓄積（nice-to-have）。

## 3. 2 択質問 → ゲーム種別（要確定）

- 質問数は **3 問**を基本（= 2³ = 8 通りの組み合わせ）。多すぎると事前生成が膨らむため。
- 各質問は遊びの軸を 1 つ決める。案（マサさん調整可）：
  - Q1: `スピード重視` ↔ `じっくり考える`
  - Q2: `攻める（破壊/得点）` ↔ `避ける（生存）`
  - Q3: `一発勝負（タイムアタック）` ↔ `積み上げ（コンボ/スコア）`
- 3 問の答え（8 通り）が、その人に授けるゲームの「型」を決める。
- ※ 質問数・内容はここを変えれば全体に反映される（SSOT）。

## 4. システム構成

```
[ブラウザ: 神社フロント web/index.html]
   │ キー操作のみ。2択の答え(3bit) → POST /api/pray {answers:["a","b","a"]}
   ▼
[神社サーバ server.py]
   │ 答え → 「型キー」(例 "spd-atk-time") に変換
   ▼
   ├─(基本) その場でライブ生成: 120b(設計)→20b(実装)→構文check を ~50秒で
   │        (ollama: gpt-oss:120b/20b @ GX10)
   └─(保険) 予算超過/失敗時のみ同型キャッシュを授与（§6.4）
   ▼
[games/<id>/index.html] を授与画面で全画面プレイ
```

## 5. コンポーネント仕様

### 5.1 神社フロント（`web/index.html` 単一ファイル）
- 画面状態：`title` / `quiz`（2択×3）/ `pray`（神社モチーフ）/ `summon`（生成待ち演出）/ `play`（iframe でゲーム）/ `result`。
- **操作は ↑↓←→ / Z / C のみ**。マウス・タップ・テキスト入力なし（C1）。
- 祈り画面：神社モチーフ表示＋ `Z` 連打/長押しで祈りゲージが溜まる演出 → 完了で生成要求。
- ゲーム表示：`<iframe>` に授かった `index.html` を読み込み全画面。

### 5.2 神社サーバ（`server.py`・既存 cockpit/server.py を拡張）
- `POST /api/pray` : body `{answers: ["a"|"b", ...]}` → 型キー算出 → 該当型のゲームを 1 本選定 → `{game_id, title, url}`。プールにあれば即返す（C2）。
- `GET /games/<game_id>/index.html` : ゲーム本体配信。
- `GET /api/games` : 生成済み一覧（センター用・nice-to-have）。
- （任意）`POST /api/generate` : バックグラウンド生成をキック（プール補充）。
- 答え→型キーの変換は純関数（テスト対象・T3）。

### 5.3 生成エンジン（`engine/` = agent-game-team 取り込み・無改造）
- `agent-game-team` を `engine/` にコピー（clean copy）。`games/` `state/` は gitignore。
- **生成時に C1/C3 を満たすため、brief に制約を必ず注入する**（§6.3）。
- 神社側は brief を渡し生成物パスを受け取るだけの薄い結合。

## 6. 生成戦略（その場でライブ生成・1 分以内に収める）

**最重要方針（マサさん指定）：AI ゲームであることが核。「その場で考える＝ライブ生成」を必ず体験に残す。**
事前のきめうちは "土台" には使うが、ゲームそのものを事前に焼かない。AI がその場でゲームを作る。

そのために、現行 `agent-game-team` の多ターン（designer→builder→runner→orchestrator ×5）を **デモ用に高速化したパイプライン**で 1 分以内に収める。

### 6.1 高速ライブ生成パイプライン（基本フロー）
1. **設計（120b・短く）**：2択3問の答え＋祈りから、ゲームの核アイデア（mechanic / 勝敗 / 操作割当）を **短い game-spec** に。出力を絞り高速化。
2. **実装（20b・1パス）**：固定スキャフォールド（§6.3）に乗せて単一 HTML を 1 回で生成。多ターンの修正ループは回さない。
3. **軽量チェックのみ**：`node --check` で JS 構文だけ確認（LLM テストループは使わない）。構文 NG なら 20b で 1 回だけ即修正。
4. 完成 → 授与。**全体を時間予算 ~50 秒に収める**（超過時は §6.4 フォールバック）。

- モデル役割：**120b = 発想（その場で考える頭脳）／20b = 実装（速い手）**。GX10 上の ollama に接続。
- 「ある程度のきめうち」= 固定スキャフォールド・操作系・終了処理（§6.3）。ゲームの中身（遊びの発明）は毎回 AI が生成する。

### 6.2 きめうちの範囲（土台のみ・ゲーム本体は焼かない）
- 固定するもの：HTML 雛形、矢印+Z+C の入力処理、3分タイマー＋結果画面の枠、ゲームループの骨。
- AI が毎回決めるもの：何のゲームか、ルール、勝敗条件、見た目、面白さの核 ←ここが「その場で考える」。

### 6.3 brief への制約注入（生成ゲームが C1/C3 を満たすため）
- 全 brief に固定で付与する制約文：
  ```
  操作は矢印キー(↑↓←→)と Z と C のみ。マウス・テキスト入力は使わない。
  1 プレイは 3 分以内に必ず終わる（制限時間 or 明確なクリア/ゲームオーバー）。
  終了時に結果画面（スコア/クリア/タイムアップ）を出し、Z で再挑戦・C でタイトルへ。
  単一 HTML・素の JavaScript・1 画面・外部ライブラリ禁止。
  ```
- 型キーごとに上記＋型の特徴（スピード/攻防/勝負形式）を足して brief を作る。

### 6.4 フォールバック（C2 を絶対に割らない安全網）
- ライブ生成が時間予算（~50 秒）を超えた/失敗した場合に備え、**直前までに生成した同タイプのゲームを少数キャッシュ**しておき、それを授与する。
- フォールバックでも「AI が生成したゲーム」である点は不変（過去にライブ生成したもの）。プールを"基本"にはせず、あくまで保険。

### 6.5 実行環境
- ハード：**GX10**（DGX Spark / GB10）。
- モデル：**gpt-oss:120b（設計）/ gpt-oss:20b（実装）**、ollama 経由。神社サーバから `OLLAMA_URL` で GX10 に接続。
- ※ 1 分以内に入るかは実機計測が前提（§10 で先頭タスク化）。収まらなければ設計出力をさらに圧縮 or 設計を JSON 化して固定ランタイムで描画する案に切替。

## 8. 受け入れ条件（テスト・完成ゲート）

- **T1**: タイトルで `Z` → 2 択質問へ。質問は ←→ 選択・`Z` 決定で進む（テキスト入力なし）。【C1】
- **T2**: 3 問回答後、祈り画面（神社モチーフ）になり、`Z` 連打/長押しで祈りが完了する。
- **T3**: 答え `["a","b","a"]` → 型キーへ変換する純関数が決定的に正しいキーを返す（単体テスト）。
- **T4**: `POST /api/pray` がその場でライブ生成したゲームを返し、授与までが **1 分以内**（実機 GX10 計測で担保）。【C2】
- **T5**: 授与ゲームが iframe 表示され、↑↓←→/Z/C で操作でき、**3 分以内に終了して結果画面**が出る。【C1/C3】
- **T6**: 生成失敗/予算超過時に同型キャッシュへフォールバックし、来場者は必ずゲームを授かれる（C2 を割らない）。

MVP 完成 = T1〜T5 が実機で 1 周（ライブ生成が 1 分に収まる）。T6 は安全網の確認。
**先頭タスク = §10-1 の実機計測**：GX10 上で高速パイプラインが何秒で 1 本作れるか測り、1 分に収まる構成を確定する。

## 付録A. 生成時間 実機計測（2026-06-13・aiko-dev）

aiko-dev 開発 PC（※GX10 ではない・参考下限値）での計測：

| 工程 | モデル | 出力 | 所要 | 速度 |
|------|--------|------|------|------|
| 設計 | gpt-oss:120b | 700 tok | **85.3 s** | ~8 tok/s |
| 実装(フルHTML) | gpt-oss:20b | 2,622 tok | **56.0 s** | 46.8 tok/s |

- 120b→20b の連続呼び出しはこの PC のメモリで **HTTP 500（モデル再読込/メモリ不足）**。両モデル常駐できない。
- **結論**：この PC では 1 分は無理。だが GX10（GB10・128GB ユニファイドメモリ）なら両モデル常駐＋高速化が見込め、下記構成なら 1 分内が狙える。
- **1分に収める構成（要 GX10 実機再計測）**：
  1. 120b は「発想」だけ・出力を ~300 tok に絞る（短い game-spec）
  2. 20b で実装 1 パス（テストループなし・構文 check のみ）
  3. 両モデルを keep_alive で常駐（コールド読込を排除）
  4. ストリーミングで生成中から「降りてくる」演出（体感待ち時間を短縮）
- 本番計測は GX10 の ollama エンドポイントで実施する（未取得）。
